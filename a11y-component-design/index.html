The Curb Cut effect

Let's talk for a second about the curb cut effect

  • One of my favorite terms, and it comes directly from disablity rights activism
  • Refers to a change championed by a small group who need it, that ends up benefitting everyone
  • Other examples: audio captions, high contrast mode/night mode, distraction-free features
  • Curb cuts
  • Audio captions and transcripts
  • High contrast mode or night mode
  • Distraction-free features
  • An accessible, semantic web?
  • Audio captions: bars, videos in public, language learners
  • high contrast mode: reading in bright environments, easier on the eyes
  • distraction-free: necessary for ADD/autism/etc, nice for everyone
  • what will be the long-term unanticipated benefits of an accesible web?

Introduce the characters

Even if you don't work with anyone like this now, I'm sure you've had the experience of working with one or more of these in the past.

Writing open source means that what you make will be exposed to all sorts of developers, and some of them will come back and submit Github issues.

Yeah, these are charicatures, but everyone has probably been in the position of wanting something to "just work", starting out and not understanding errors, and needing to extend default functionality before. They're all valid use cases, and it's reductive and unhelpful to dismiss all poor code usage as user error.

Additionally, coding for the most extreme cases will make the average better

“If it looks right, I'm done.”
“What's an ARIA? I don't like opera.”
“I need these twenty obscure features, and a pony.”
Lack of accessibility is not malicious. Make it easy, and they will come around.

Common button use cases:

    type="submit | reset"
  • Open a panel

Think of as many valid permutations as you can, and handle them well. Don't expect the end developer to think about semantic patterns or non-mouse interaction well

Who really knows best?

  • a11y is interesting in that it's not an area where people generally have experience, and is often done wrong
  • It isn't just me, this matters for anyone making tools that can be used on the front end. People will use your stuff!
  • Sure you can think of other areas you know best -- you designed your code for a reason, after all
  • Even though these slides use charicatures, they are real use cases -- someone shouldn't need to think too hard to do a basic use case (Caveman), it should be as helpful and intuitive in-editor as possible (interns), and there will always be valid use cases you haven't thought of at design-time (inventor)

Build extensibility into:

  • Visual styles
  • Layout
  • Child content

Provide an intuitive API for built-in defaults:


							interface ButtonProperties {
							  type: 'submit' | 'reset' | 'button' | 'toggle'
							        | 'menu' | 'accordion';
							  disabled: boolean;
							  pressed: boolean;
							  open: boolean;
							}

Including menu/accordion with button types allows you to set aria-haspopup and aria-expanded behind the scenes

Alternatively, have separate menu buttons and accordion buttons

The limits of a framework

You will never be able to:

  • Write appropriate labels and alt text
  • Take care of overall page structure (landmarks, heading hierarchy)
  • Account for all possible use cases

Be realistic: a framework can never do all the work, and shouldn't claim to. There are plenty of other a11y requirements you can't handle in a library

Context is everything! There is so much guesswork around what covers "most" use cases. There are always outliers

Instead:

  • Be realistic up front
  • Provide good usage documentation
  • Educate other developers about diverse interaction patterns
